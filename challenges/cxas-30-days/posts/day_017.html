Detailed Lesson: Online Evals & Latency in CXAS

Once your agent is in production, the work doesn't stop. You need to monitor how it performs with real users. **Online Evaluations** involve analyzing live traffic data to assess quality and performance. Unlike offline testing with golden datasets, online evals show you the messy reality of user interactions.

One of the most critical aspects of online performance is **Latency**—the time it takes for the agent to respond to a user input. In CXAS, we typically look at two key latency metrics:

  • p50 (Median Latency): The value at which 50% of requests are faster and 50% are slower. This gives you a good sense of the "typical" user experience.
  • p99 Latency: The value at which 99% of requests are faster. This highlights the worst-case scenarios and edge cases that are affecting a small but significant portion of your users. High p99 latency often indicates issues with complex queries, large context sizes, or slow external tools.

Analyzing Trace Reports

To fix high latency, you need to know where the time is being spent. CXAS provides **Trace Reports** that break down a single conversation turn into its component steps. A typical trace might include:

  1. Pre-processing: PII redaction, guardrail checks.
  2. LLM Processing (Input): Loading context, processing system instructions and history.
  3. Tool Execution: Time spent waiting for external APIs or databases to return data.
  4. LLM Generation (Output): The time taken by the model to generate the response tokens.

The 'Why' from the LLM Perspective

Why do some turns take so much longer than others? From the LLM's perspective, latency is heavily influenced by two factors:

  • Context Size: LLMs process information by attending to all tokens in their current context window. As the conversation grows longer, or if large amounts of data are injected via tools, the model has to do more computation just to process the input, increasing latency.
  • Tool Execution: If your agent relies on tools to fetch external data (e.g., checking order status), the LLM must stop, emit the tool call, wait for the external system to respond, and then process that new information. The latency of the tool call is often the largest contributor to the overall turn latency.
Understanding this helps you design better prompts and tools. For instance, using tool wrappers to combine multiple API calls can reduce the number of round trips the LLM needs to make, significantly cutting down latency.

Pro-Tip: Perceived Latency vs. Absolute Latency

Sometimes you can't make the backend faster. In those cases, focus on **perceived latency**. Use **Streaming** to start showing the user the response as it is being generated token-by-token, rather than waiting for the full response to be complete. This makes the agent *feel* much faster even if the total time is the same.

Watch-Out: Ignoring the Long Tail

Don't fall into the trap of only looking at average or p50 latency. A great p50 might hide terrible p99 latency. If 1% of your users are experiencing 15-second delays, those users are likely having a very poor experience that could lead to escalation or churn. Always monitor and optimize for the long tail.
