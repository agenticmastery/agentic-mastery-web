Lesson: Simulator & Tracing in CXAS

Building an agent is only half the battle. The other half is testing and debugging. In CX Agent Studio, the **Simulator** is your interactive sandbox where you can chat with your agent in real-time to see how it responds.

But simply seeing the response isn't enough when things go wrong. That's where **Execution Tracing** comes in. For every turn in the Simulator, CXAS generates a detailed trace that allows you to inspect the internal operations of the agent.

How to Read a Trace

A full execution trace typically reveals the following stages:

  1. Prompt Assembly: You can see the final prompt sent to the LLM, including your instructions, few-shot examples, and any retrieved context from tools like Data Stores.
  2. Model Output (Reasoning): You see the raw output from the model. This often includes the model's "Chain of Thought" or internal reasoning before it decides to take an action.
  3. Tool Calls: If the model decides it needs external information, the trace shows the specific tool it selected and the arguments it generated for that tool call.
  4. Tool Execution: You see the output returned by the tool (e.g., the search results from a Data Store).
  5. Final Response: The final text that is sent back to the user.

By examining these steps, you can see exactly where a breakdown occurred. Did the prompt lack the right instructions? Did the model fail to call the tool? Or did the tool return the wrong data?

The 'Why' from the LLM Perspective

Why is tracing so critical? From the LLM's perspective, every response is a result of calculating probabilities based on the input context (the prompt and conversation history). The "attention mechanism" determines which parts of the input the model focuses on. When an agent fails (e.g., hallucinates, calls the wrong tool, or gives a bad answer), it's usually because the attention was misdirected by ambiguous instructions or conflicting context. Tracing allows us to inspect the intermediate steps and see exactly where the attention failed. Without tracing, an LLM is a black box. With tracing, it's a glass box.

The Day 3 Challenge: Debug the Trace

Scenario: EcoShop's return agent is failing in production. A customer asked about returning a damaged item. The agent should have checked the policy using the return_policy_store tool, but instead, it gave a generic apology and said it couldn't help.

The Trace Data:

[User]: "I received a broken item yesterday. How do I return it?"

[Internal Trace - Prompt]:
...
<instructions>
You are a helpful customer service agent for EcoShop.
When a customer asks about returning a standard item, use the tool {@TOOL: return_policy_store} to check the policy.
Be polite and concise.
</instructions>
...

[Internal Trace - Model Output]:
"I apologize that your item arrived broken. I cannot help you with returns directly. Please contact human support."

[Internal Trace - Tool Calls]:
None.

[Agent Response]: "I apologize that your item arrived broken. I cannot help you with returns directly. Please contact human support."

Task: 1. Analyze the trace provided above. 2. Identify why the agent failed to call the return_policy_store tool. 3. Propose a specific update to the instructions in the prompt to fix this failure so that it handles "broken" or "damaged" items correctly using the tool.

Note: The solution guide with the correct analysis and prompt fix will be linked in tomorrow's lesson! Do not give away the solution.

Difficulty: Medium. Requires reading the trace and understanding how LLM instructions map to actions.

Pro-Tip: Best Practice

When writing instructions for tools, use broad semantic triggers rather than narrow keywords. Instead of just "Use tool for returns," use "Use tool for returns, refunds, damaged items, or exchanges." This helps the LLM generalize better across different user phrasings.

Watch-Out: Pitfall

A common pitfall is looking only at the final output when debugging. If the agent gives a wrong answer, always check the trace to see if the tool was called and what data it returned. The LLM might be fine, but the data retrieved might be missing the answer.
