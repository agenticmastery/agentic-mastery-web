Lesson: OpenAPI & Connector Tools in CXAS

To make an agent useful in the real world, it needs to interact with external systems—CRMs, ERPs, databases, and custom APIs. In CX Agent Studio (CXAS), these capabilities are exposed to the agent as **Tools**.

There are two primary ways to connect external APIs in CXAS:

  • OpenAPI Tools: You provide an OpenAPI specification (formerly Swagger) that describes your API's endpoints, methods, parameters, and schemas. CXAS uses this spec to understand how to call your API.
  • Pre-built Connectors: For common enterprise systems like Salesforce, ServiceNow, or Jira, CXAS provides pre-built connectors that simplify the authentication and configuration process, allowing you to quickly expose specific functionalities as tools.

Defining Tools with OpenAPI

An OpenAPI spec is a JSON or YAML document that describes a RESTful API. When you create an OpenAPI tool in CXAS, you upload this spec. The platform parses it and creates a tool for each operation defined in the spec.

To make a tool available to an agent, you must explicitly attach it to the agent's configuration. Once attached, the agent can decide to call it based on the conversation context.

The 'Why' from the LLM Perspective

How does the LLM know which tool to call and what parameters to pass? The LLM doesn't actually "execute" the API call. Instead, the OpenAPI specification is provided to the LLM as part of its context. When the LLM determines that a tool is needed to fulfill a Customer User Journey (CUJ), it generates a special structured output (a tool call) containing the name of the operation and the required arguments, mapped from the user's input.

This is why **descriptions** in your OpenAPI spec are critical. The LLM relies on the summary, description, and parameter descriptions to understand *what* the endpoint does and *when* to use it. If your descriptions are vague, the LLM may fail to call the tool or pass the wrong parameters.

Using Pre-built Connectors

For systems like Salesforce, instead of writing a full OpenAPI spec for the entire Salesforce API, you can use a CXAS Connector. You authenticate with the service, and then select the specific actions (e.g., "Create Lead", "Get Account Details") you want to expose as tools to your agent. This drastically reduces development time and maintenance overhead.

The Day 9 Challenge: Define the Order Status Tool

Scenario: EcoShop wants to enable customers to check the status of their orders. They have a REST API with an endpoint GET /orders/{order_id} that returns order details.

Task: Define a mock OpenAPI spec snippet (in YAML or JSON) for this endpoint. Your definition must include: 1. A clear summary and description so the LLM knows when to use it. 2. The order_id parameter, specified as a path parameter, with a description that helps the LLM extract it from user input. 3. A successful response (200 OK) schema containing at least status and delivery_date fields.

Do not give away the full implementation, but focus on providing the structure that the LLM would need to understand this tool.

Difficulty: Medium. Focus on writing clear descriptions for the LLM.

Pro-Tip: Be Explicit in Descriptions

Treat parameter descriptions in your OpenAPI spec as instructions for the LLM. For example, instead of just saying "Order ID", say "The 10-digit alphanumeric order ID provided by the customer, usually starting with 'ES'." This helps the LLM extract the correct value reliably.

Watch-Out: Avoid Exposing Too Many Tools

While it's tempting to expose every API endpoint as a tool, doing so can confuse the LLM and increase latency. Only expose the specific tools needed for the Customer User Journeys (CUJs) that the agent is designed to handle.
