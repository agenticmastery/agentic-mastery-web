Challenge Recap

In the Day 17 challenge, you were asked to analyze a sample trace report for a slow EcoShop interaction (5.2s total latency) and identify which step was the primary bottleneck.

Recommended Solution: Trace Analysis & Optimization

1. Bottleneck Identification

Based on the trace report provided in the challenge:

  • Step 1: Pre-processing (0.1s) - Negligible.
  • Step 2: LLM Input Processing (0.4s) - Reasonable for context loading.
  • Step 3: Tool Execution (3.9s) - This is the clear bottleneck. It accounts for 75% of the total turn latency.
  • Step 4: LLM Output Generation (0.8s) - Normal for generating the response.

Conclusion: The optimization effort must focus on the {@TOOL: lookup_order} tool execution. Optimizing LLM prompts or generation length will yield minimal improvements compared to fixing this tool call.

2. Recommendations for Optimization

To address this bottleneck, consider the following strategies:

  • API Optimization: Work with the backend team to optimize the API endpoint used by {@TOOL: lookup_order}. Ensure proper database indexing and minimize payload size.
  • Caching: If order status data doesn't change second-by-second, implement caching for frequent lookups.
  • Tool Wrappers: If the tool makes multiple sequential API calls internally, wrap them into a single optimized call or execute them in parallel.

3. Structured Instructions for Latency Awareness

While instructions can't make the API faster, they can guide the LLM to handle the interaction gracefully, such as setting expectations or using streaming if supported. Here is an example of how to structure instructions for a CUJ involving a slow tool:


<subtask name="Order Status Lookup">
    <step number="1">
        <trigger>User asks for order status and provides an order number.</trigger>
        <action>
            Inform the user that you are looking up the order details (to manage perceived latency).
            Call the tool {@TOOL: lookup_order} with the provided order number.
        </action>
    </step>
    <step number="2">
        <trigger>Tool returns order data.</trigger>
        <action>
            Synthesize the response for the user, highlighting the current status and expected delivery date.
        </action>
    </step>
</subtask>

Notice how Step 1 explicitly manages the user's expectation before the slow tool call is made, which is a best practice for improving user experience during high-latency operations.
