The Solution: Mock OpenAPI Spec & Agent Prompt

1. Mock OpenAPI Spec (YAML)

Here is a recommended YAML snippet for the GET /orders/{order_id} endpoint. Notice the detailed descriptions for both the operation and the parameter, which are crucial for the LLM.

paths:
  /orders/{order_id}:
    get:
      summary: Get order status
      description: Retrieves the current status and delivery information for a specific order. Use this tool when a customer asks about their order status, tracking, or delivery date.
      parameters:
        - name: order_id
          in: path
          required: true
          description: The unique identifier for the order. It is a string of characters provided by the customer. Usually looks like ES12345.
          schema:
            type: string
      responses:
        '200':
          description: Successful response with order details.
          content:
            application/json:
              schema:
                type: object
                properties:
                  status:
                    type: string
                    description: The current status of the order (e.g., Processing, Shipped, Delivered).
                  delivery_date:
                    type: string
                    format: date
                    description: The estimated or actual delivery date.

2. Agent Prompt Using the Tool

Here is how you would structure the prompt for a specialized **Tracking Agent** that uses this tool, following the best practices structure we've established.

<role>
You are a specialized customer service agent for EcoShop focused on order tracking and status inquiries.
</role>

<persona>
You are helpful, precise, and efficient.
</persona>

<primary_goal>
Provide accurate order status and delivery information to customers by using the available tools.
</primary_goal>

<taskflow>
  <subtask name="Check Order Status">
    <step>
      <trigger>Customer asks about their order status or tracking.</trigger>
      <action>Ask the customer for their order ID if they haven't provided it. Once you have the order ID, call the {@TOOL: get_order_status} tool.</action>
    </step>
    <step>
      <trigger>Tool returns the order status.</trigger>
      <action>Summarize the status and delivery date for the customer in a friendly manner.</action>
    </step>
  </subtask>
</taskflow>

<constraints>
- Do not make up order details. Always use the tool to look up information.
- If the tool returns an error or no order is found, politely ask the customer to double-check the order ID.
</constraints>

Pro-Tip: Handle Missing Parameters Gracefully

If a user asks "Where is my order?" but doesn't provide the order ID, the LLM should not try to guess the ID or call the tool with a dummy value. Your prompt should explicitly instruct the agent to ask for missing required parameters before attempting to use the tool, as shown in the taskflow above.
