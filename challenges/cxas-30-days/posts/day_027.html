Detailed Lesson: Session Testing with SCRAPI

The Sessions client in SCRAPI allows you to simulate user interactions with your App programmatically. This is crucial for building automated test suites that run as part of your CI/CD pipeline, ensuring that updates to prompts or tools don't break existing Customer User Journeys (CUJs).

1. Initializing the Sessions Client

To get started, you need to import and initialize the Sessions client.

from cxas_scrapi import Sessions

sessions_client = Sessions()
APP_NAME = "projects/your-project-id/locations/your-location/apps/your-app-id"

2. Text Testing

Sending a text query is the most common operation. You use the run method, passing the app name, the user text, and a session ID.

response = sessions_client.run(
    app_name=APP_NAME,
    text="I want to check the status of my order 12345",
    session_id="test-session-unique-id"
)

print(f"Agent Response: {response.text}")
# You can now assert on the response text
assert "status" in response.text.lower()

Using a consistent session_id allows you to maintain context across multiple turns in a single test case. To start a fresh conversation, use a new session_id.

3. Audio Testing

SCRAPI also supports testing audio modality, which is essential for validating voice-enabled agents. You can pass audio bytes and set the modality to "audio".

with open("query_audio.wav", "rb") as f:
    audio_content = f.read()

response = sessions_client.run(
    app_name=APP_NAME,
    audio=audio_content,
    modality="audio",
    session_id="test-voice-session-id"
)

print(f"Agent Response (Text from Audio): {response.text}")

The "Why" from the LLM Perspective

Why do we need automated script-based testing when LLMs are so smart? Because **LLMs are non-deterministic**. A prompt that works perfectly 95% of the time might fail on the 96th turn due to a slight variation in the user's phrasing or internal temperature sampling. By automating tests and running them repeatedly (or across a wide batch of inputs), we can reveal these non-deterministic failures before they impact customers. It allows us to measure confidence intervals for specific Customer User Journeys (CUJs) rather than relying on "vibes" from a few manual tests.

The Day 27 Challenge: Automated Regression Test Script

Scenario: EcoShop wants to ensure that their agent always correctly handles the "Check Order Status" Customer User Journey (CUJ). You need to write a script to automate this check.

Your Task: Write a Python script using the CXAS SCRAPI library to:

  1. Initialize the Sessions client.
  2. Define a list of 3 different ways a user might ask to check their order status (e.g., "Where is my stuff?", "Order status please", "Track order 999").
  3. Loop through these queries, sending each to the agent.
  4. Verify that the agent's response contains relevant words like "order" or "status" or asks for an order number.
  5. Print a clear report showing which queries passed and which failed.

Focus on structure and assertions. Do not hardcode the solution script here.

Pro-Tip: Test Isolation

Always use a unique, randomly generated session ID for each independent test case. This ensures that the state from a previous test doesn't bleed into the current one, preventing false positives or negatives.

Watch-Out: Rate Limiting

When running batch tests programmatically, be mindful of the API rate limits for your project. Rapidly firing hundreds of queries might trigger throttling. Implement exponential backoff in your test scripts if you encounter 429 Too Many Requests errors.
