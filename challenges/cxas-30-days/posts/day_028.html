Detailed Lesson: Programmatic Simulation Evals

While session testing is excellent for regression checks on specific phrases, Simulation Evals use an LLM-driven "User Simulator" to interact with your agent. This allows you to test how your agent handles context, multi-turn logic, and unexpected user behaviors across full Customer User Journeys (CUJs).

1. The Evaluations Client

SCRAPI provides the Evaluations client to manage these complex tests. Instead of sending raw text, you define a Persona and a Goal for the simulator.

from cxas_scrapi import Evaluations

evals_client = Evaluations()
APP_NAME = "projects/your-project-id/locations/your-location/apps/your-app-id"

2. Triggering a Simulation

To trigger a simulation, you define the configuration, including the persona (who the user is) and the success criteria (what the agent must achieve or prevent).

simulation_config = {
    "app_name": APP_NAME,
    "user_persona": "A customer who received a damaged item and is very upset. They do not have the original packaging.",
    "conversation_goal": "Insist on a full refund and a shipping label for return.",
    "max_turns": 8,
    "success_criteria": "The agent must offer a replacement or refund and provide instructions for return without packaging, while maintaining a empathetic tone."
}

print("Triggering simulation eval...")
result = evals_client.run_simulation(simulation_config)

3. Analyzing Results

The result object contains detailed information about the simulation, including whether it passed the success criteria, a score, and the full conversation transcript.

print(f"Status: {result.status}") # e.g., PASSED or FAILED
print(f"Score: {result.score}")
print(f"Reasoning: {result.reasoning}")

if result.status == "FAILED":
    print("Conversation Log:")
    for turn in result.conversation_log:
        print(f"{turn.role}: {turn.text}")

The "Why" from the LLM Perspective

Why rely on simulations rather than just large datasets of real logs? Because simulations allow you to **explore the edge cases** that haven't happened yet. By defining diverse, adversarial, or complex personas, you force the agent into state spaces it might rarely visit in production. Automated quality gates powered by these simulations ensure that a change to improve one CUJ doesn't silently break another complex multi-turn journey. It provides continuous assurance in a non-deterministic environment.

The Day 28 Challenge: Scripted Simulation Eval

Scenario: EcoShop wants to ensure that their agent strictly follows the return policy: items must be returned within 30 days. They want to simulate a customer trying to return an item purchased 45 days ago.

Your Task: Write a Python script using the CXAS SCRAPI library to:

  1. Initialize the Evaluations client.
  2. Define a simulation configuration for the "Late Return" scenario.
    • Persona: A polite but firm customer trying to return shoes bought 45 days ago.
    • Goal: Get a refund despite being outside the 30-day window.
    • Success Criteria: The agent must politely refuse the refund based on the policy but offer an alternative like store credit or an exchange.
  3. Trigger the simulation.
  4. Programmatically check if the status is "PASSED".
  5. Print the reasoning provided by the evaluation engine.

Focus on the script structure and assertions. Do not give away the solution script here!

Pro-Tip: Persona Versioning

Treat your evaluation personas as code. Store them in version control along with your agent configurations. As your product evolves, your test personas should evolve too. This ensures your test suite remains relevant.

Watch-Out: Simulation Costs

Because simulation evals use LLMs for both the user side and the evaluation logic, they are more resource-intensive and costly than simple unit tests. Run them strategically—e.g., on a schedule or before major releases—rather than on every single commit.
