Solution: Optimizing EcoShop's Return Prompt

Here is the proposed solution to fix the failure where the agent ignored the 48-hour constraint for damaged items.

We will use the structured PIF format (Prompt Instructions Format) as defined in our best practices, focusing on the <subtask> and <step> tags to guide the model's attention.

<subtask>Handle Returns for Damaged Items</subtask>
<step>
<trigger>User states they want to return an item because it arrived damaged or defective.</trigger>
<action>
1. Ask the user for their order number if they haven't provided it.
2. Once you have the order number, call the tool {@TOOL: get_order_details} with the order_id.
3. **Crucial Constraint Check:** From the tool output, find the delivery date.
4. Calculate the time elapsed between the delivery date and the current date.
5. **Decision Path:**
   - **If more than 48 hours have passed:** Politely inform the user that damaged items must be reported within 48 hours of delivery for a full refund. Offer to check if they qualify for a standard return (if within 30 days and item is unused) or store credit.
   - **If 48 hours or less have passed:** Proceed with processing the return for a full refund. Ask for any additional details needed (e.g., photos of damage) as per standard flow.
</action>
</step>

Why This Fix Works

  • Explicit Trigger: We isolated the specific condition ("item arrived damaged") to ensure the model pays attention to this specific flow.
  • Step-by-Step Logic: We broke down the action into sequential steps, making it easier for the model to follow reasoning paths.
  • Crucial Constraint Callout: We used strong language like "**Crucial Constraint Check**" and bolded text (which can help in some prompt formats, or translated to strong imperative text) to draw attention.
  • Explicit Decision Path: By mapping out the "If/Else" logic clearly, we reduce the chance of the model making a bad inference.

Pro-Tip: Use XML-like tags for complex logic

When handling multi-step reasoning or strict constraints, using XML-like tags (like <step>, <trigger>, <action>) helps the model parse the structure better than a simple block of bullet points. It provides a pseudo-code structure that LLMs are generally good at following.
