Detailed Lesson: One-Click Deploy Orchestration

Manual configuration is the enemy of scale. If you have to click in a UI to connect a tool or manually trigger a test, you cannot guarantee that what you tested is what is actually deployed. Programmatic orchestration ensures that your agent's configuration is versioned code.

1. The Orchestration Flow

A full automated build typically follows these steps:

  1. Create/Update Agent: Use the Apps client to define the agent's core prompt and settings.
  2. Create/Bind Tools: Use the Tools client to register functions and attach them to the agent.
  3. Baseline Testing: Use the Sessions or Evaluations client to run a set of standard tests to ensure the new build is healthy.

2. Combining the Clients

Here is how you can structure a script to combine these operations using SCRAPI:

from cxas_scrapi import Apps, Tools, Evaluations
import os

def full_automated_build():
    project_id = os.environ.get("GCP_PROJECT_ID")
    location = "global"
    
    apps_client = Apps()
    tools_client = Tools()
    evals_client = Evaluations()

    # Step 1: Create the Agent
    print("Creating agent...")
    app_config = {
        "display_name": "EcoShop Automated Agent",
        "instructions": "You are a helpful assistant for EcoShop. Use tools when necessary.",
        "model": "gemini-1.5-pro"
    }
    app = apps_client.create_app(project_id, location, app_config)
    app_id = app.name.split("/")[-1]
    print(f"Agent created with ID: {app_id}")

    # Step 2: Add a Tool
    print("Registering tool...")
    tool_config = {
        "display_name": "CheckStock",
        "description": "Check stock for a given item.",
        "function_spec": {
             "name": "check_stock",
             "parameters": {
                 "type": "OBJECT",
                 "properties": {
                     "item_name": {"type": "STRING"}
                 },
                 "required": ["item_name"]
             }
        }
    }
    tool = tools_client.create_tool(project_id, location, tool_config)
    
    # Bind tool to app (conceptual)
    print("Binding tool to agent...")
    apps_client.bind_tool(app.name, tool.name)

    # Step 3: Run Baseline Test
    print("Running baseline test...")
    simulation_config = {
        "app_name": app.name,
        "user_persona": "A regular customer asking about stock.",
        "conversation_goal": "Ask if organic apples are in stock.",
        "max_turns": 3,
        "success_criteria": "The agent must attempt to use the {@TOOL: CheckStock} tool."
    }
    result = evals_client.run_simulation(simulation_config)
    print(f"Test Status: {result.status}")

if __name__ == "__main__":
    full_automated_build()

The "Why" from the LLM Perspective

From an LLM's perspective, environment consistency is everything. If the prompt says the agent has access to a tool called CheckStock, but that tool was not correctly bound in the backend due to a manual error, the LLM will still try to invoke it, leading to a runtime failure that it cannot recover from. Complete automation minimizes this "configuration drift." It ensures that the agent's internal model of its capabilities perfectly matches the reality of the infrastructure it is deployed on, leading to much more predictable and reliable behavior.

The Day 29 Challenge: EcoShop One-Click Deploy

Scenario: EcoShop wants a single script that they can run in their CI/CD pipeline to deploy a brand new instance of their agent whenever they update the core prompt or tools.

Your Task: Write a Python script using the CXAS SCRAPI library to automate the following sequence:

  1. Create a new app named "EcoShop-FullAuto-[YourInitials]".
  2. Create a tool named {@TOOL: GetOrderDetails} with a simple function specification (requiring an `order_id`).
  3. Bind that tool to the newly created app.
  4. Run a short simulation eval to verify that the agent correctly identifies when to use the tool (e.g., when asked "Where is my order?").
  5. Print the final result of the build and test.

Do not hardcode project IDs; use environment variables. Focus on the orchestration logic. Do not give away the solution script here!

Pro-Tip: Idempotency

Make your build scripts idempotent. This means if you run the script twice, it shouldn't fail or create duplicate resources. Check if a tool or app already exists before creating it, or use an "update or create" logic. This makes your automation much more robust.

Watch-Out: Cleanup on Failure

If your automated build fails halfway (e.g., the tool creation succeeds but the test fails), ensure your script handles cleanup. Leaving orphaned apps or tools around can clutter your project and increase costs. Use try...finally blocks to clean up resources if the build process does not complete successfully.
