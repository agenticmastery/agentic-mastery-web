Detailed Lesson: PII Redaction & DLP in CXAS

In CX Agent Studio, protecting user privacy is a top priority. **Personally Identifiable Information (PII)** includes any data that could potentially identify a specific individual, such as names, email addresses, phone numbers, and credit card details. **Data Loss Prevention (DLP)** refers to the strategies and tools used to ensure that sensitive data is not lost, misused, or accessed by unauthorized users.

CXAS provides built-in features to automatically detect and redact PII from user inputs before they are processed by the Large Language Model (LLM). This ensures that sensitive data is never stored in conversation logs in its raw form and is not used to influence model behavior in ways that could leak information.

Configuring Redaction Rules

To implement effective PII redaction, you typically configure rules based on:

  • **Pre-defined Detectors:** Many systems offer built-in detectors for common PII types like credit card numbers, social security numbers, and email addresses.
  • **Custom Regular Expressions (Regex):** For domain-specific sensitive data (e.g., a specific format for EcoShop loyalty IDs), you can define custom regex patterns to identify and redact matching text.
  • **Masking Strategies:** You can choose how to replace the sensitive data. Common strategies include replacing it with a generic label (e.g., `[REDACTED_EMAIL]`) or masking parts of it (e.g., `****-****-****-1234`).

The 'Why' from the LLM Perspective

Why is it critical to redact PII *before* it reaches the LLM's core context? LLMs operate by attending to all the tokens in their context window. If a raw credit card number or email address is included in the input prompt, it becomes part of the model's active memory for that turn. While modern LLMs are generally safe, there are risks: the data could be inadvertently echoed back in the response, logged in plain text in system logs, or potentially exposed if the conversation history is used for future fine-tuning or debugging. By redacting at the gateway (pre-processing), we ensure the model never "sees" the raw PII, adhering to the principle of least privilege and minimizing the blast radius of any potential data leak.

Pro-Tip: Use Context-Aware Redaction

While simple regex works for structured data like credit cards, some PII like names can be tricky. Over-redaction can make the conversation hard for the LLM to understand. Where possible, use context-aware or entity-recognition-based redaction that understands the difference between a common word and a person's name in context.

Watch-Out: Redaction Breakage in Tool Calls

Ensure that if the user *needs* to provide PII for a valid operation (like an email for lookup), that data is handled securely via a specific, secure tool call or payload that bypasses the general LLM context logging, rather than being blindly redacted if the LLM needs to pass it to a tool. If you redact it before the LLM sees it, the LLM cannot pass it to the tool! This is why some systems use "reversible" redaction or tokenization for trusted tools.
