Detailed Lesson: Offline Evaluations in CXAS

In traditional software, we use unit and integration tests. In the world of LLMs and Agentic systems, testing is less deterministic. Users can say the same thing in a million different ways, and the LLM's response can vary. To ensure quality, we use **Offline Evaluations**.

Offline Evals run test cases against a snapshot of your agent (or specific components) without exposing it to real users. In CXAS, we typically look at two types of offline evaluations:

1. Turn Evaluations

Turn Evals focus on a single exchange: a user utterance and the agent's immediate response or action. You are testing if the agent understood the user's intent (mapped to the correct CUJ) and took the right next step.

For example, if the user says "I want to return this broken toaster," a Turn Eval checks if the agent correctly identifies the return flow and perhaps calls the `{@TOOL: query_policy}` tool, rather than trying to sell them a new toaster.

2. Simulation Evaluations

Simulation Evals are more complex. They simulate a complete, multi-turn conversation between a user persona (played by another LLM or a script) and your agent. The goal is to see if the agent can guide the user to a successful resolution of a full Customer User Journey (CUJ).

For instance, a Simulation Eval might test a user trying to return a product, being told it's past the return window, and then asking for an exception. The evaluation checks if the agent followed the policy throughout the conversation and arrived at the correct final state.

Creating Golden Datasets

To run these evaluations, you need a **Golden Dataset**. This is a curated, high-quality set of test cases that represent the "ground truth" of how your agent should behave. It should cover:

  • **Happy Paths:** Standard, straightforward user journeys.
  • **Edge Cases:** Unusual but valid user requests (e.g., returning a gift without a receipt).
  • **Negative Cases:** Things the agent should *not* do (e.g., processing a return for a non-returnable item).

A typical test case in a Golden Dataset includes the input (user utterance or persona), the expected output (agent response or tool call), and success criteria.

The 'Why' from the LLM Perspective

Why can't we just use traditional unit tests? LLMs are non-deterministic by nature. The same input can produce slightly different outputs. Traditional testing looking for exact string matches will fail often. Offline evaluations using LLMs as judges or measuring semantic similarity allow us to scale quality assurance. We can run hundreds of simulated conversations in minutes to detect regressions that a human tester might miss, ensuring that improvements in one area don't degrade performance in another.

The Day 13 Challenge: Define a Golden Dataset for Returns

Scenario: You are building the Golden Dataset for the EcoShop Return Assistant. You need to define test cases to ensure the agent handles the return process correctly.

Include at least:

  • One Turn Eval test case for initial return request.
  • One Simulation Eval test case for a complete return flow.

Specify the Input, Expected Tool Call (using `{@TOOL: tool_name}` syntax), and Expected Final State for each.

Pro-Tip: Diversify Your Expressions

When creating Turn Evals, don't just use the most common phrasing. Include slang, typos, and indirect requests to ensure your agent can handle the variety of real-world user input. A robust Golden Dataset reflects the messy reality of human conversation.

Watch-Out: Overfitting to the Golden Set

A common pitfall is tuning your prompts specifically to pass the test cases in your Golden Dataset. This is called overfitting. If you do this, your agent might perform perfectly on the test set but fail on new, slightly different real-world inputs. Keep your dataset representative and update it regularly with real user interactions that exposed weaknesses.
