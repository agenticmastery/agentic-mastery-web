Detailed Lesson: Isolation Testing of Tools and Callbacks

In complex Agentic systems, the agent is just the orchestrator. The actual work—fetching data, updating systems, sending notifications—is often done by **Tools** and **Callbacks**. If something goes wrong, you need to know exactly where the failure occurred. Was it the LLM failing to call the tool? Or was it the tool itself returning incorrect data?

**Isolation Testing** (similar to unit testing in traditional software) allows you to validate these components independently of the LLM. This makes tests faster, cheaper, and much easier to debug.

Validating Tools in Isolation

Tools are typically functions or API wrappers called by the agent. To test a tool in isolation, you bypass the LLM entirely. You call the tool function directly with specific inputs and verify that it returns the expected output.

For example, consider a tool `{@TOOL: calculate_shipping}`. To test it in isolation:

  • **Input:** A specific destination zip code and package weight.
  • **Execution:** Call the tool function directly.
  • **Verification:** Check if the returned shipping cost matches the expected value based on the business logic or a mock API response.

If the tool relies on external APIs, you should **mock** those API responses to ensure your tests are deterministic and don't rely on external system availability.

Validating Callbacks in Isolation

Callbacks are hooks that trigger when specific events occur in the agent's lifecycle (e.g., `on_tool_start`, `on_agent_finish`, or custom business events). They are crucial for logging, state management, and side effects.

To test a callback in isolation, you simulate the event that triggers it and verify that the callback performs the correct action.

For example, suppose you have a callback that flags high-value returns for manual review. To test it:

  • **Trigger:** Manually invoke the callback function, passing in a mock return request where the value is over the threshold (e.g., $100).
  • **Verification:** Verify that the callback successfully calls the alerting system or updates the database state to "flagged".

The 'Why' from the LLM Perspective

Why is component-level isolation testing so critical when working with LLMs? LLMs are non-deterministic and operate as "black boxes" to some extent. If you only perform end-to-end testing, a failure could be due to prompt ambiguity, LLM hallucination, a bug in the tool logic, or a failure in the callback execution. By unit testing tools and callbacks in isolation, you dramatically reduce the search space for bugs. You ensure that the deterministic parts of your system are perfect, leaving you to focus your LLM evaluation efforts on the non-deterministic conversational aspects.

The Day 14 Challenge: Isolation Test Plan for EcoShop

Scenario: EcoShop uses a tool called `{@TOOL: apply_loyalty_discount}` which takes a `customer_id` and the current `cart_total`, and returns the new total after applying any applicable discounts based on the customer's loyalty tier.

Task: Write a test plan to validate the `{@TOOL: apply_loyalty_discount}` tool in isolation. Do not write the actual code, but define the test cases in a structured way.

Your test plan should include at least three test cases covering:

  • A standard case (e.g., a "Gold" tier customer getting a 10% discount).
  • An edge case (e.g., a customer with no loyalty tier or an invalid ID).
  • A case where the cart total is zero or negative.

For each test case, specify the **Input Parameters** (customer_id, cart_total), the **Mock Data** needed (what the mock loyalty service should return for that ID), and the **Expected Output** (the calculated final total).

Pro-Tip: Use Standard Unit Testing Frameworks

You don't need special AI testing tools for isolation testing. Use standard frameworks like `pytest` (Python) or `Jest` (JavaScript). Treat your tools and callbacks as regular functions. Mocking libraries (like `unittest.mock` in Python) are your best friends here for simulating external API responses.

Watch-Out: Isolation Is Not Integration

A common mistake is assuming that because all tools and callbacks pass their isolation tests, the agent will work perfectly. Isolation testing ensures the *components* work as specified. You still need Turn and Simulation Evals to ensure the LLM *calls* the tools correctly and understands their outputs in the context of a Customer User Journey (CUJ).
