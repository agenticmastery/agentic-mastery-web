Lesson: Data Stores & RAG in CXAS

Large Language Models (LLMs) are incredibly powerful, but they only know what they were trained on. They don't know your company's specific policies or real-time data. To solve this, we use **Retrieval-Augmented Generation (RAG)**.

The Pre-requisite: Vertex AI Agent Builder

Here is a critical technical detail many get wrong: **You do not create the Data Store content inside CX Agent Studio.**

Data Stores are part of the broader Google Cloud ecosystem (specifically **Vertex AI Agent Builder**, formerly Generative AI App Builder). The workflow is as follows:

  1. You first create and ingest data into a Data Store in the **Vertex AI Search and Conversation** console.
  2. Once the data is ingested and indexed there, you then go to **CX Agent Studio** and create a **Tool** that references that existing Data Store.

This separation ensures that data management and conversational logic are kept separate, allowing you to update documents without breaking the agent's prompt.

Other Tool Types in CXAS

While Data Stores are powerful for your own data, CX Agent Studio also supports other tool types to extend capabilities:

  • Google Search Tool: Allows the agent to perform live Google searches to answer general knowledge questions or find public information.
  • File Search: For searching content directly within uploaded files without setting up a full website crawl.

Mentor Tip: Real-World Practice
You don't need a live corporate website to master Data Stores. You can practice by indexing any public documentation site or a specific Wikipedia page. Better yet, simulate an enterprise setup by uploading a mock return policy document to a Google Cloud Storage bucket and pointing your Data Store there. This is exactly how real-world enterprise implementations are built!

The 'Why' from the LLM Perspective

Why do we do this? From the perspective of the LLM, it transforms the task from "recall from memory" to "open-book exam." LLMs are prone to hallucination when they try to remember specific facts. By injecting the precise, relevant context directly into the prompt (via the Data Store retrieval), we reduce the need for the model to guess. The attention mechanism focuses on the provided context, leading to dramatically higher accuracy.

The Day 2 Challenge: Knowledge-Grounded Return Agent

Scenario: EcoShop has stored its return policy in a document. They no longer want the rules hardcoded in the prompt. They want you to use a Data Store tool.

Task: 1. First, refer to the official documentation on how to create and ingest data into a Data Store in Vertex AI Agent Builder: Create Data Store Documentation. (You don't need to create one if you don't have a project, but understand the flow). 2. Now, update the EcoShop Return Agent prompt from Day 1. Instead of hardcoding the 30-day rule, instruct the agent to use a Data Store tool named return_policy_store to find the answers. 3. In the instructions for the return Customer User Journey (CUJ), explicitly reference the tool using the correct syntax: {@TOOL: return_policy_store}. 4. Do NOT hardcode the policy details in the prompt. Let the tool handle that.

Note: The solution guide with the correct prompt structure and tool reference will be linked in tomorrow's lesson!

Difficulty: Medium. Focus on correct tool referencing syntax and understanding the data store creation flow.

Pro-Tip: Best Practice

When referencing tools in instructions, be specific about *why* and *when* the agent should use them. Instead of just saying "Use the tool," say "Use {@TOOL: return_policy_store} to find the return window and eligibility criteria." This helps the model make better decisions.

Watch-Out: Pitfall

A common pitfall is trying to create the Data Store directly inside the agent flow. Remember to follow the top-down approach: Create in Agent Builder first, then reference in CXAS.
