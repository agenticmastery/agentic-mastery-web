Lesson: Tool Chaining & Wrappers

In complex Customer User Journeys (CUJs), an agent often needs to perform multiple operations to satisfy a single user request. For example, to answer "Is this item available and when can I get it?", the agent might need to:

  1. Call a tool to check inventory.
  2. Call another tool to calculate shipping time based on the user's location.

If the agent handles this by calling the first tool, processing the result, and then calling the second tool, it takes two full "turns" (Model -> Tool -> Model). Each turn adds latency, making the experience feel sluggish to the customer.

Tool Chaining

Tool chaining refers to the ability of the agent to plan and execute a sequence of tool calls. While CXAS handles orchestration, you can optimize this by providing clear instructions on how tools relate to each other, or by using parallel tool calling if the platform supports it and the calls are independent.

The Wrapper Pattern

A more powerful way to reduce latency is to create a **Wrapper Tool**. Instead of exposing two separate, fine-grained tools to the agent, you create a single, coarse-grained tool that combines both operations on the backend.

For example, instead of check_inventory(item_id) and get_shipping_estimate(zip_code), you could create a wrapper tool called get_product_availability_and_delivery(item_id, zip_code). This tool executes both checks in a single execution turn from the agent's perspective, drastically reducing latency.

The 'Why' from the LLM Perspective

Reducing the number of execution turns doesn't just improve speed; it also improves the reliability of the agent. Every time the model has to stop, generate a tool call, and then process the tool output, it adds to the conversation history (context window).

By minimizing turns, we keep the context window smaller and more focused. This helps the attention mechanism of the LLM stay focused on the user's primary goal, reducing the risk of the model getting confused by intermediate outputs or hallucinating across turns. It makes the agent "smarter" simply by making it more efficient.

Pro-Tip: Balance Granularity

While wrappers reduce latency, don't over-consolidate. Keep tools focused enough that they are reusable across different CUJs. A tool that does too many unrelated things will confuse the LLM about when to use it.

Watch-Out: Hidden Backend Latency

A wrapper tool reduces turns between the LLM and the tool execution environment, but if the wrapper itself calls slow external APIs sequentially, total latency might still be high. Ensure your backend implementation of the wrapper executes independent operations in parallel whenever possible.
