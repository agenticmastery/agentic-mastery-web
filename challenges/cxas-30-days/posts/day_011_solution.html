Day 11 Solution: Tool Chaining & Wrappers

Explore the optimized designs for reducing latency through tool wrappers and efficient instruction flows.

Super Mentor Guidance

Motivation: Excellent work on tackling Day 11! Latency is the silent killer of good user experience in conversational AI. By thinking about how to minimize turns, you are moving from a functional agent to a high-performance agent.

Solution: EcoShop Unified Availability Check

Here are two ways to solve the Day 11 challenge. Option 1 is generally preferred for maximum latency reduction, while Option 2 is useful if you cannot change the existing tool implementations.

Option 1: The Wrapper Tool Approach (Recommended)

We create a new, combined tool that handles both operations on the backend. This reduces the interaction to a single turn.

Tool Name: get_product_fulfillment_details

Description: Checks if a product is in stock and calculates the estimated delivery time to a specific zip code.

Parameters:

  • product_id (string): The ID of the product to check.
  • zip_code (string): The destination zip code for shipping.

Returns: A JSON object containing in_stock (boolean) and estimated_days (integer).

This wrapper would be implemented on your server. When called, it can execute the database check for stock and the API call for shipping estimation in parallel, further reducing latency.

Option 2: The Instruction Set Approach (Tool Chaining)

If you must keep the tools separate (e.g., {@TOOL: check_stock} and {@TOOL: calculate_delivery}), you can instruct the model to chain them efficiently. We use structured prompting to guide the model.

Optimized Prompt Structure

You are a helpful assistant for EcoShop. When a customer asks about product availability and delivery time, follow these steps:

<subtask name="check_full_availability">
  <step number="1">
    Identify the product_id and the customer's zip_code from the conversation.
  </step>
  <step number="2">
    Call {@TOOL: check_stock} with the product_id.
  </step>
  <step number="3">
    If the item is in stock, immediately call {@TOOL: calculate_delivery} using the zip_code. Do not wait to report the stock status before calling the second tool.
  </step>
  <step number="4">
    Provide a unified answer to the customer containing both availability and delivery time.
  </step>
</subtask>

If the item is out of stock in Step 2, skip Step 3 and inform the customer it is out of stock.

This structure uses the <subtask> and <step> tags to clearly define the sequence, helping the model understand it should execute these as a continuous flow rather than waiting for user input between steps.
