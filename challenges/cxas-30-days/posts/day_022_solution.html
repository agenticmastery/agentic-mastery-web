Day 22 Solution: DFCX to CXAS Migration (Legacy Transition)

Solution and guidance for mapping legacy Dialogflow CX flows to CXAS instructions.

Super Mentor Guidance

Motivation: You've successfully translated a rigid state machine into a fluid, goal-oriented instruction block! This is the core skill required to modernize conversational AI. By focusing on the user's goal rather than drawing complex state graphs, you unlock the true power of generative agents.

Achievement: You have mapped multiple pages and transitions from a legacy Dialogflow CX flow into a single, structured Customer User Journey (CUJ) in CXAS.

Solution: Migrating EcoShop's Cancellation Flow

Here is the proposed solution to translate the legacy Dialogflow CX "Cancel Order" flow into a structured CXAS instruction block for a Customer User Journey.

<subtask>Cancel Order</subtask>
<step>
<trigger>User expresses a desire to cancel an order.</trigger>
<action>
1. Ask the user for their order number if they have not already provided it.
2. Once the order number is obtained, call the tool {@TOOL: get_order_status} using the provided order_id.
3. **Decision Path (Order Status):**
   - **If the status is 'shipped':** Inform the customer that the order has already shipped and unfortunately cannot be canceled. The journey ends here.
   - **If the status is 'pending':** Proceed to the next step.
4. Ask the customer to confirm the cancellation (e.g., "Are you sure you want to cancel order #12345?").
5. **Decision Path (Confirmation):**
   - **If the user confirms (yes):** Call the tool {@TOOL: cancel_order} with the order_id. Inform the user that the order has been successfully canceled.
   - **If the user does not confirm (no):** Inform the user that the order will not be canceled and will proceed as scheduled.
</action>
</step>

Why This Fix Works

  • Goal-Oriented Structure: We condensed a multi-page Dialogflow CX flow into a single sequential list of steps within the instruction block. The model uses its context memory to track progress rather than hardcoded state changes.
  • Explicit Tooling: We used the correct syntax for calling tools ({@TOOL: tool_name}), explicitly telling the agent when to fetch external data or perform actions.
  • Clear Decision Logic: We modeled the legacy "Transitions" as explicit if/else checks within the instruction block. This gives the LLM clear guidelines on how to branch based on tool output and user confirmation, without restricting its ability to interpret natural variations in how users say "yes" or "no".
  • Eliminated Hardcoded Input Categories: Notice that we don't define rigid categories for user input like "providing order ID" or "saying yes". The LLM infers these implicitly from context and the instructions provided.

Pro-Tip: Use the instruction format as your new state machine

While we avoid complex graphs, the structured use of <subtask>, <step>, and clear condition checks acts as a pseudo-state machine that keeps the LLM on track without limiting its natural language capabilities.
