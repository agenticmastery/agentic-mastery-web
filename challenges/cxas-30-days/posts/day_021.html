Detailed Lesson: Manual Prompt Optimization

Even the best-designed agents will fail on edge cases. Automated evaluations (like the ones we discussed in Week 3) are excellent for identifying *where* the agent fails, but they often don't tell you *why* or *how* to fix it. That is where manual prompt optimization comes in.

The process follows a continuous loop:

  1. Analyze Failure Traces: When an evaluation fails, look at the complete conversation trace. Pay attention to:
    • What the user said.
    • What tools the agent called and the outputs returned.
    • The agent's final response.
  2. Identify the Root Cause: Is it a **reasoning error** (the model failed to deduce the correct next step based on available information) or a **constraint violation** (the model ignored a specific rule or instruction)?
  3. Apply Targeted Optimization: Update the prompt instructions to address the specific failure mode. Avoid adding a massive new section for every edge case; instead, refine existing instructions or use structured tags to improve clarity.
  4. Re-Evaluate: Run the evaluation again to ensure the fix works and hasn't introduced regressions in other CUJs.

The 'Why' from the LLM Perspective

Why does changing a few words in a prompt often fix a complex reasoning error? It comes down to **Attention Mechanisms**. LLMs process tokens by calculating attention scores between them. If instructions are buried in a large block of text, the model's attention may be diluted.

When you manually optimize a prompt by breaking it into structured tags (like <constraints> or <subtask>) or by using imperative, unambiguous language, you are effectively guiding the model's attention. You are increasing the weights assigned to critical rules and reducing the probability of the model getting "lost" in the context.

The Day 21 Challenge: Optimizing EcoShop's Return Policy

Scenario: EcoShop has a strict return policy: *Damaged items must be reported within 48 hours of delivery for a full refund.* Standard returns (not damaged) are allowed within 30 days.

In a recent evaluation run for the "Process Return" CUJ, the agent failed on a specific turn. Here is the trace:

User: "I want to return this shirt I bought last week. It has a tear."
Agent: "I can help with that. What is your order number?"
User: "It's #98765."
[Agent calls tool: get_order_details(order_id='98765')]
[Tool returns: Order #98765. Delivered 5 days ago.]
Agent: "Okay, I see your order was delivered 5 days ago. Let's proceed with your return. I've generated a prepaid label for you."

The Failure: The agent recognized the item was damaged ("has a tear") and that it was delivered 5 days ago, but it proceeded with the return anyway, violating the 48-hour constraint for damaged items.

Your Task: Tweak the prompt instructions to fix this specific failure. Propose a new or updated <subtask> or <constraints> block that ensures the agent checks the delivery timeframe for damaged items before proceeding. Do not give away the solution! Do not link the solution at the bottom.

Pro-Tip: Change One Thing at a Time

When optimizing prompts manually, treat it like a scientific experiment. Change only one instruction or constraint at a time and re-run your evals. If you change five things at once and the performance improves (or degrades), you won't know which change was responsible.

Watch-Out: Instruction Overload

Avoid the temptation to add a new rule for every single edge case you find. This leads to "instruction overload," where the prompt becomes too long and the model starts ignoring rules again. Always look to refine existing instructions before adding new ones.
