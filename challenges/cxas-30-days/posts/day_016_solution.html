Recommended Solution: PII Redaction Configuration

Here is a recommended configuration for the redaction rules in EcoShop. This setup ensures that sensitive data is masked before processing, while maintaining the ability to handle specific user journeys securely.

Configured Redaction Rules

PII Type Detection Approach Replacement Token Rationale
**Credit Card Number** Regex pattern matching standard 16-digit (or 15-digit for Amex) credit card formats, including spacing variations. `[REDACTED_CC]` Direct exposure of CC numbers is a high risk. Immediate full masking is required for PCI compliance.
**Email Address** Standard email regex pattern detection. `[REDACTED_EMAIL]` Protects user identity. However, as noted in the lesson, if the email is needed for a lookup tool, a reversible tokenization approach is preferred.

Structured Instructions for Secure PII Handling

To implement a secure pattern where the LLM needs to reference PII without seeing it (Reversible Redaction), you can use structured instructions like this:


<subtask name="Secure Order Lookup">
    <step number="1">Identify if the user wants to look up an order and needs to provide an email address.</step>
    <step number="2">If the user provides an email, check if it has been replaced by a session token like `[SESSION_EMAIL_1]` by the pre-processor.</step>
    <step number="3">Call the tool {@TOOL: lookup_order} using the session token `[SESSION_EMAIL_1]` as the identifier. Do not attempt to extract or use the raw email address.</step>
</subtask>

This keeps the PII out of the LLM's attention and logs, while still enabling the necessary business logic by letting the tool execution layer handle the actual data swap.
